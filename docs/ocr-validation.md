# Bedrock OCR 検証ログ

最終更新: 2026-04-16
関連: [implementation-plan.md](./implementation-plan.md) / [design.md](./design.md) / [requirements.md](./requirements.md)
参考: [画像前処理検討フェーズ](./画像前処理検討フェーズ%20344152c9ab9680558f81fc4a62475375.md)

このドキュメントは Bedrock による ISHIDA ITB 計量機 7 セグ表示の OCR 検証を
**時系列で蓄積** していく生きたログです。画面(PWA)→ S3 → Lambda → Bedrock の
本番経路を実際に通しながら、精度・挙動・改善点を記録していきます。

検証は **画面経由(E2E)で一本化** しており、プロンプトやモデルを変えたくなった場合は
[amplify/functions/ocr-handler/handler.ts](../amplify/functions/ocr-handler/handler.ts) を
編集して `npx ampx sandbox` で再デプロイします。

---

## 1. 目的と判定基準

### 目的
1. Bedrock の Claude 系モデルが ISHIDA ITB の 7 セグ LED を安定して読めるかを確認する
2. 画面(PWA)→ S3 → Lambda → Bedrock の本番パイプラインが疎通することを確認する
3. プロンプトと Tool use 定義を現場画像に合わせて調整する
4. 画像前処理(Sharp / OpenCV)の効果を段階的に検証する
5. 現場運用時の操作性(撮影しやすさ・結果の見やすさ・誤読時の補正)をレビューする

### 判定基準(実装計画書より)
- ✅ **正解率 95% 以上** → 本実装決定、現行設定を確定版として固定
- ⚠️ **80–95%** → 前処理(コントラスト強調・トリミング UI)を追加して再評価
- ❌ **80% 未満** → 方針見直し(専用 OCR / 計量機シリアル直結 / ハイブリッド)

正解率は「完全一致」と「±目量以内(例: 0.05 kg)」の 2 基準で算出する。
集計は手作業(スプレッドシート等)で行う。

### 判定に必要なサンプル数の目安
| 段階 | 枚数 | できる判断 |
|------|------|----------|
| 初期 | 3〜4 枚 | モデル傾向差、プロンプト感度、前処理の効果切り分け、経路の疎通 |
| 中期 | 10 枚 | プロンプトの優劣比較(粗い精度判定) |
| 本検証 | 20 枚以上 | 正解率の統計的判定(95%/80% の判定基準に耐える) |

## 2. 検証方法(画面経由 E2E)

本番経路([Measure.tsx](../apps/tablet/src/pages/Measure.tsx) → S3 → AppSync Mutation
`invokeOcr` → Lambda → Bedrock)を実際に通す方式です。精度評価と
本番パイプラインの疎通確認を同時に行います。

### 前提
- `npx ampx sandbox` が稼働していること(Lambda・AppSync・Storage が立っている)
- Cognito テストユーザーが 1 人以上登録済み
- Bedrock のモデルアクセスが有効化済み
- [Measure.tsx](../apps/tablet/src/pages/Measure.tsx) が動くこと(既に M4 で実装済み)

### 手順
1. `npm run dev:tablet` でタブレットアプリをローカル起動(または PC のブラウザで開く)
2. Cognito ユーザーでログイン
3. `/containers` で検証用の仮容器を 1 件登録(まだ無ければ)
4. `/measure` で画像を 1 枚ずつ投入
5. 結果パネルを確認し、以下を実験ログに記録:
   - OCR 値(`value`)、信頼度(`confidence`)、安定マーク(`stable`)
   - 警告(`warnings`)、`rawText`
   - **体感レイテンシ**(OCR 実行ボタン → 結果表示まで)
6. 画像ごとの期待値と照らし合わせ、完全一致/±目量以内で採点
7. 結果を §4 実験ログに追記

### プロンプト/モデル/前処理を変更する場合
1. [handler.ts](../amplify/functions/ocr-handler/handler.ts) を編集
2. `npx ampx sandbox` 稼働中ならホットリロードで反映される
3. 何を変えたかを §4 の実験ログに記録

## 2.1 前処理パイプラインの段階的検証計画

精度・レイテンシの改善を **1 サブステップずつ追加検証** する。
各ステップで同じ 4 枚(`samples/initials/`)を投入し、前ステップとの差分を記録する。
十分な精度が出た時点でそれ以降のステップは実施しない。

### 改善ステップ一覧

```
Step 1  Haiku 切替(前処理なし)         ✅ 実施済み(Exp#2)
  ↓ 精度不十分
Step 2-1  Sharp: リサイズのみ           ← 次
  ↓ 精度不十分なら
Step 2-2  Sharp: + グレースケール + normalize
  ↓ 精度不十分なら
Step 2-3  Sharp: + sharpen
  ↓ 精度不十分なら
Step 3    OpenCV: グレア除去 + CLAHE + ROI 自動検出(別計画)
```

### Step 2-1: リサイズのみ

**仮説**: 4080×3072(≒16,000 トークン)→ 長辺 800px(≒640 トークン)に縮小するだけで
トークン数が 1/25 になり、レイテンシ改善 + モデルが表示部に集中しやすくなる。

**変更**:
- `sharp` を npm install
- [handler.ts](../amplify/functions/ocr-handler/handler.ts) に `preprocessImage` 関数を追加
- リサイズ(長辺 800px, `fit: 'inside'`) → JPEG quality 90 で出力

**検証**: 4 枚投入。特に 55.50(3↔5 混同)が改善するかを確認。

### Step 2-2: + グレースケール + コントラスト強調

**仮説**: LED の明暗をグレースケールで際立たせ、`normalize()`(ヒストグラムストレッチ)で
コントラストを強調すると、反射でぼやけたセグメントの判別が改善する。

**変更**:
- パイプラインに `.grayscale()` と `.normalize()` を追加
- Bedrock に送る `media_type` を `image/jpeg` に固定

**検証**: 4 枚投入。2-1 との差分で効果を確認。

### Step 2-3: + シャープネス強調

**仮説**: 桁の輪郭をくっきりさせ、セグメントの ON/OFF 判別精度を上げる。

**変更**:
- パイプラインに `.sharpen()` を追加

**検証**: 4 枚投入。2-2 との差分で効果を確認。
過剰なシャープネスでノイズが増える場合はパラメータ(sigma)を調整。

### Step 2 全体の判定基準

| 結果 | 次のアクション |
|------|-------------|
| 4 枚正解 + レイテンシ改善 | Step 2 で完了。本番画像待ち |
| 55.50 が改善(3→5 の混同が解消) | パラメータ微調整して確定 |
| 55.50 が依然誤読 | Step 3(OpenCV: グレア除去 + CLAHE)へ |
| 前処理で他の画像が悪化 | パラメータを緩めるか、その処理をスキップ |

### Sharp の限界(Step 3 が必要になるケース)

Sharp にはできない処理がある。以下が必要になったら Step 3(OpenCV)に移行する:

| 処理 | Sharp | OpenCV |
|------|-------|--------|
| CLAHE(適応的コントラスト強調) | ❌ | ✅ |
| グレア除去(HSV ベース + インペイント) | ❌ | ✅ |
| ROI 自動検出(表示部の輪郭抽出) | ❌ | ✅ |
| テンプレートマッチング | ❌ | ✅ |

特に **グレア(光反射)が 3↔5 誤読の根本原因**
([参考資料 §3](./画像前処理検討フェーズ%20344152c9ab9680558f81fc4a62475375.md))なら、
Sharp の `normalize()` では解決できず、HSV ベースのグレア除去(OpenCV)が必要になる。
