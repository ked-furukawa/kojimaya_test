# そば粉製造 計量ミス防止システム 要件定義書

最終更新: 2026-04-21

## 1. 背景・課題

- そば粉製造工程において、原材料の計量ミスが年に数回発生している。
- ミスにより製品廃棄が発生し、損失額は **年間数十万円規模**。
- 現在は作業者個人の注意・確認に依存しており、属人的な仕組みとなっている。
- 「人が間違える前提」で、**仕組み(システム)で検知・防止する** 方式へ切り替えたい。

## 2. システムの目的

- 原材料の計量結果を客観的に記録し、生産計画上の正しい値と自動照合する。
- 誤った計量を **作業確定前に検知** し、廃棄ロスをゼロに近づける。
- 作業履歴を残し、トレーサビリティおよび改善活動の基礎データとする。

## 3. ユーザー像 / 利用環境

| 項目 | 内容 |
|------|------|
| 主な利用者 | 製造現場の作業者(複数名) |
| 利用デバイス | タブレット(現場設置 / 持ち運び) |
| 利用場所 | 工場内の計量エリア(粉塵あり・濡れ手NG想定) |
| ネットワーク | 工場内Wi-Fi(オフライン耐性は要相談) |

## 4. 計量機器の仕様(現物より)

- メーカー: ISHIDA
- 型式系: ITB
- ひょう量切替: 30kg / 60kg / 150kg
- 目量: 0.01kg / 0.02kg / 0.05kg
- 表示部: 7セグメント数字 + 単位「kg」、安定マーク、ゼロ点、風袋表示などのインジケータあり
- データ出力(シリアル/RS232等)があるかは未確認 → **本システムでは画像OCR方式を採用**

## 5. 業務フロー(想定)

1. 事前準備
   - 生産管理担当が **Excelの生産計画** をシステムに取り込む
   - 各レシピの「材料」「目標質量」「許容誤差(±)」が登録される
2. 現場作業
   1. 作業者がタブレットで本日の生産計画(ロット/レシピ)を選択
   2. 一覧から計量する材料を選択
   3. 計量機の表示部をタブレットカメラで撮影
   4. システムがOCRで質量値を抽出 → 数値を画面に表示
   5. 目標値 ± 許容誤差と自動照合
      - **OK** : 「OK」表示、記録して次の材料へ
      - **NG** : 「過多/不足」を赤表示、計量やり直しを促す(確定不可)
3. 完了
   - 全材料がOKになった時点でロットを「計量完了」として記録
   - 結果はクラウドに保存され、後から照会・出力可能

## 6. 機能要件

### 6.0 機能一覧(現場ヒアリングより)

| # | 機能 | 概要 |
|---|------|------|
| A | **指示重量の計算** | 現行の Excel 相当のロジック。**乾麺 / 生麺の2パターン** に対応 |
| B | **指示重量の表示** | 作業者に見やすい形(大きい文字・色分け)で提示 |
| C | **OCRによる重量読み取り** | 重量計の液晶表示をカメラで撮影し、画像から数値を抽出 |
| D | **一致判定** | 指示重量と OCR 値の照合(許容誤差込み) |
| E | **加水率の記録・調整による再計算** | PoCでは**加水率(%)の記録**のみ実装。再計算は計算式確定後(後続フェーズ) |
| F | **初回計量の風袋引き(複数個対応)** | 同一容器を複数個まとめて使う運用を想定し、使用個数を指定して風袋を差し引く |
| G | **計量履歴の保存** | 全工程の記録をクラウドに保存 |

### 6.1 マスタ管理
- [F-01] Excel(生産計画)のインポート
  - 列例: ロットNo / 製品名 / 材料名 / 目標質量(kg) / 許容誤差(kg または %) / 順序
  - 取り込み時にバリデーション(必須項目・数値範囲・重複)
- [F-02] 材料マスタ・製品マスタ(必要に応じて)
- [F-03] ユーザー(作業者)管理 ※ Amplify Auth で実装
- [F-04] **容器(風袋)マスタ**
  - 現場で使用する容器が **概ね決まっている** ため、登録済み容器から選択する運用
  - 各容器に「容器名 / 風袋重量(kg) / 用途メモ / 既定フラグ」を保持
  - 1件を **デフォルト容器** として指定可能(初回計量時に自動選択)
  - **ハードコードせず**、画面から追加・編集・無効化が可能
  - 容器を変更した履歴は監査ログとして保持(過去のロットは登録時の風袋値を保持)
  - マスタは**1個あたりの風袋重量**を保持する。同一容器を**複数個同時に使う**運用を想定し、使用個数は計量時に指定する(F-15 参照)
- [F-05] **レシピ(配合)マスタ**
  - 製品ごとに「材料 / 基準配合比 / 基準加水率」を保持
  - **乾麺 / 生麺** 2系統のテンプレートを持てる構造
  - レシピの改訂時はバージョンを上げ、過去ロットは旧版を参照

### 6.2 指示重量の計算
- [F-08] **指示重量計算エンジン**
  - 入力: 製品/レシピ、ロットサイズ(打数 or 仕上げkg)、麺種別(乾麺/生麺)、加水率
  - 出力: 各材料の必要質量(kg)
  - 乾麺/生麺で計算式を切り替え
- [F-09a] **加水率の記録(PoCスコープ)**
  - 計量時に加水率(%)を **任意入力** として記録可能
  - 入力範囲 0〜100、小数第1位想定(※運用ルール確定は [questions.md](./questions.md) A-2 待ち)
  - 未入力のまま保存してもよい(計算連動が未実装のため必須化しない)
  - 値は各計量記録に **スナップショットとして保持**
- [F-09b] **加水率変更時の即時再計算(後続フェーズ)**
  - F-08 の計算式確定後に着手
  - 現場判断で加水率を変更した場合、影響を受ける材料の指示重量を即時再計算
  - 変更前/変更後の値を履歴に残す(誰が・いつ・なぜ変更)

### 6.3 計量・OCR
- [F-10] タブレットカメラから計量機の表示部を撮影
- [F-11] 撮影画像から **質量値(数値+小数+単位)** を抽出
- [F-12] OCR結果の信頼度が低い / 安定マーク未確認 の場合は再撮影を促す
- [F-13] 作業者による手動補正入力(OCR誤読の救済策)
- [F-14] 撮影画像はエビデンスとしてクラウド保存
- [F-15] **初回計量時の風袋引き(複数個対応)**
  - 容器マスタからデフォルト容器を自動選択(現場で変更可)
  - **使用個数** を指定(既定 1、0 以上の整数)
  - **正味重量 = 計量値 − (風袋重量 × 使用個数)**、として判定に使用
  - 使用個数 = 0 は「風袋引きしない(= 計量値 = 正味重量)」を意味する
  - 容器ID / 風袋重量 / 使用個数 はロット記録に **スナップショットとして保存**(後から容器マスタが更新されても過去ロットの値は変わらない)

### 6.4 照合・判定
- [F-20] 指示重量 ± 許容誤差 で自動判定(OK / 過多 / 不足)
- [F-21] NG時は「あと何kg足す/減らす」を表示
- [F-22] 全材料完了で「ロット完了」確定
- [F-23] **指示重量の見やすい表示**
  - 大きい数字、OK/NGの色分け、目標との差分バー

### 6.5 履歴・トレーサビリティ
- [F-30] ロット単位の計量履歴閲覧(誰が・いつ・何kgを記録したか)
- [F-31] 撮影画像との紐付け
- [F-32] CSV/Excel エクスポート

## 7. 非機能要件

| 区分 | 要件 |
|------|------|
| 性能 | OCR応答 概ね 3 秒以内 / 撮影~判定まで 5 秒以内を目標 |
| 可用性 | 業務時間中(平日日中)に稼働。月次メンテは要相談 |
| セキュリティ | 認証必須(Amplify Auth)、生産情報は社外公開しない |
| 端末 | iPad / Android タブレットの主要ブラウザで動作(PWA想定) |
| 保守性 | レシピ追加・許容誤差変更は管理者が画面から実施可能 |
| ログ | 操作ログ・OCR結果・補正履歴を保持 |

## 8. OCR方式の選定方針(検討事項)

候補:
1. **クラウドOCR(AWS Textract / Rekognition Custom Labels / Bedrock Vision)**
   - 精度高、開発容易、オフライン不可、利用料あり
2. **オンデバイス(Tesseract.js 等)**
   - オフライン可、精度は要チューニング
3. **ハイブリッド**
   - まずオンデバイスで前処理→クラウドで補正

7セグメントは一般的なOCRが苦手なため、**Bedrock(Claude等のVLM)** または **専用前処理 + Textract** が現実解と想定。
→ PoC では Bedrock Vision でまず精度検証する案を推奨。

## 9. 想定リスク・確認事項

未確定事項は別紙 [questions.md](./questions.md) に集約。

### 9.1 確定済みの方針(2026-04-21時点)
- 既存Excelの計算式は **未入手**。まずは **OCR機能の実装を先行** させ、計算ロジックは後続フェーズで対応する。
- 加水率の業務ルール・計算連動は未ヒアリングだが、**データ登録(F-09a)は先行** してPoCから蓄積する(計算連動は後続)。
- 容器は **同一種類×複数個** を同時使用する運用がある(例: バケツ 8 個)。個数指定を計量画面の入力項目として持つ(F-15)。
- 工場Wi-Fi は安定しているため、**オンライン前提の設計** で進める(オフライン耐性は当面スコープ外)。

## 10. スコープ外(初期PoCでは扱わない)

- 計量機との直接通信(シリアル等)
- 在庫管理・発注連携
- 製造実績の会計システム連携
- 多工場・多ライン対応(まずは1拠点)
